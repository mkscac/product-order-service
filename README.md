# Единое техническое задание

## Отрабатываемый материал

Основы .NET, генераторы, `IEnumerable`, `IAsyncEnumerable`, `IEnumerator`, `IDisposable`, `async/await`, `TaskCompletionSource`, `CancellationToken`, каналы, потоки, `Parallel`, HTTP, Refit, `Microsoft.Extensions.Configuration`, `ConfigurationProvider`, `Microsoft.Extensions.Options`, `Npgsql`, PostgreSQL, serialization, ASP.NET, hosted service, HTTP, gRPC, реактивное межсервисное взаимодействие, Kafka

## Задачи

Реализовать backend-систему управления товарами и заказами, состоящую из нескольких связанных между собой сервисов и модулей

Система должна включать

- сервис конфигурации отвечающий за получение и обновление конфигураций из внешнего сервиса
- инфраструктурный слой отвечающий за работу с базой данных миграциями и репозиториями
- application слой содержащий основную бизнес-логику сервиса товаров и заказов
- gRPC presentation слой предоставляющий API над бизнес-логикой сервиса
- Kafka presentation слой отвечающий за публикацию и обработку событий
- HTTP gateway предоставляющий HTTP API над gRPC сервисом
- общие механизмы работы с коллекциями асинхронным взаимодействием и обработкой сообщений

## Функциональные требования

### Configuration

Сервис конфигураций должен предоставлять функционал

- получения конфигураций из внешнего сервиса конфигураций
- поддержки двух реализаций клиента
  - с ручной реализацией HTTP-запросов
  - с использованием `Refit`
- пагинированной выгрузки конфигураций
- хранения конфигураций в кастомном `ConfigurationProvider`
- обновления конфигураций через отдельный обслуживающий сервис
- периодического обновления конфигураций
- отказа от reload конфигурации в случае отсутствия изменений
- передачи конфигураций в остальные части системы через `Microsoft.Extensions.Configuration` и `Microsoft.Extensions.Options`
- поддержки динамического изменения конфигураций во время работы приложения

### Application

Application слой должен предоставлять функционал

- создания товаров
- пагинированного поиска товаров с фильтрацией по
  - идентификаторам
  - минимальной цене
  - максимальной цене
  - подстроке в имени товара
- создания заказа
- добавления товаров в заказ
- удаления товаров из заказа
- перевода заказа в статус `processing`
- отмены заказа
- завершения заказа по результатам обработки внешним сервисом
- пагинированного поиска заказов с фильтрацией по
  - идентификаторам
  - статусу
  - автору
- пагинированного поиска по позициям заказов
- пагинированного поиска по истории одного заказа
- добавления записей в историю заказа при всех бизнес-операциях
- ограничения операций с заказом в зависимости от его статуса
- реализации бизнес-логики в гексагональной архитектуре

### Infrastructure

Инфраструктурный слой должен предоставлять функционал

- подключения к PostgreSQL
- регистрации подключений к базе данных через DI
- регистрации и запуска миграций через `FluentMigrator` и SQL код
- реализации репозитория товаров заказов позиций заказа истории заказов
- выполнения SQL-запросов через `Npgsql`
- работы с транзакциями для операций включающих несколько запросов в базу данных
- полиморфной сериализации и десериализации записей истории заказа
- параметризации конфигурации подключения к базе данных через Options
- получения конфигурации базы данных из сервиса конфигурации

### Presentation.Kafka

Kafka presentation слой должен предоставлять функционал

- публикации событий в топик `order_creation`
- чтения событий из топика `order_processing`
- публикации события о создании заказа
- публикации события о переводе заказа в статус `processing`
- обработки событий жизненного цикла заказа приходящих из внешнего сервиса
- перевода заказа в статус `cancelled` при получении неуспешных событий
- перевода заказа в статус `completed` при получении события об успешном завершении доставки
- записи данных о событиях жизненного цикла заказа в историю
- интеграции с application слоем без дублирования бизнес-логики

### Kafka

Kafka сервис должен предоставлять функционал

- общих абстракций для записи сообщений в Kafka
- общих абстракций для чтения сообщений из Kafka
- выделения интерфейса producer
- выделения интерфейса обработчика сообщений consumer
- реализации фонового сервиса вычитывания сообщений
- батчовой обработки сообщений consumer-ом
- параметризации подключения названий топиков и размеров батчей через Options
- регистрации Kafka компонентов через методы расширения
- независимости producer и consumer от конкретных топиков
- использования Kafka как механизма реактивного межсервисного взаимодействия

### Presentation.gRPC

gRPC presentation слой должен предоставлять функционал

- публикации gRPC API над сервисом заказов
- публикации gRPC API над операциями создания и поиска сущностей
- предоставления всего функционала реализованного application слоем
- использования отдельных gRPC контрактов
- поддержки полиморфных моделей через `oneof`
- возврата ошибок в едином формате через серверный `Interceptor`
- получения конфигурации клиента через Options
- получения начальной конфигурации и её последующей актуализации через ASP.NET

### Gateway

HTTP gateway должен предоставлять функционал

- HTTP API над gRPC сервисом
- маршрутизации HTTP-запросов к gRPC presentation слою
- публикации Swagger документации
- аннотирования возможных кодов ответа
- поддержки полиморфных HTTP моделей через наследование
- корректного отображения полиморфных моделей в Swagger
- преобразования gRPC исключений в HTTP коды через middleware
- предоставления эндпоинтов сервиса заказов
- предоставления эндпоинтов связанных с интеграцией сервиса обработки заказа
- реализации HTTP API в соответствии с REST-конвенциями

### Общие механизмы

Общие механизмы системы должны предоставлять функционал

- реализации async/await-обёртки над библиотекой с реактивной моделью взаимодействия
- корректной поддержки отмены через `CancellationToken`
- поддержки батчинга сообщений
- поддержки завершения обработки сообщений

## Нефункциональные требования

- гексагональная архитектура
- проект должен быть реализован как единая система из нескольких модулей
- все конфигурации должны быть параметризуемыми
- все зависимости должны регистрироваться через DI
- код должен быть расширяемым и не должен содержать жёсткой связанности между слоями
- application слой не должен зависеть от конкретных деталей инфраструктуры и presentation слоя
- инфраструктурный слой не должен содержать бизнес-логику
- Kafka producer и consumer не должны быть привязаны к конкретным топикам
- код в `Program.cs` сервисов не должен явно выполнять бизнес-операции и должен состоять только из настройки `builder.Services`, `builder.Configuration`, `app`
- для работы с базой данных должны использоваться SQL-запросы и `Npgsql`
- для миграций должны использоваться `FluentMigrator` и SQL код
- для конфигураций должны использоваться `Microsoft.Extensions.Configuration` и `Microsoft.Extensions.Options`
- для HTTP клиента внешнего сервиса должна быть реализована версия через `IHttpClientFactory`
- для HTTP клиента внешнего сервиса должна быть реализована версия через Refit
- для Kafka интеграции должны быть реализованы общие интерфейсы и переиспользуемые компоненты
- реализация должна поддерживать динамическое обновление конфигурации
- реализация должна поддерживать unit- и интеграционные сценарии тестирования
